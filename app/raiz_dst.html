    1 · Overview & Purpose

    The RAIZ Farm Analysis DST (Decision Support Tool) is a fully offline, browser-based tool for Extension Service Agents working with smallholder farmers in agroecological intensification projects.

    It converts a farm management model into a field-usable interface that allows agents to:

    • Define a local farming context (crops, calendar, labour, cropping systems) once as a Project
    • Record individual household farm data as Farm Profiles
    • Calculate food security, economic resilience, and labour balance for each farm
    • Simulate alternative crop plans and compare them with the current situation (Scenarios)
    • Export farm data as CSV or JSON for centralised research analysis, directly from the Sessions tab

    The tool runs entirely in the browser. No internet connection is needed after opening the file. No data is sent to any server.

    Navigation — 7 tabs

    2 · Data Storage & Sessions

    Auto-save (browser storage)

    Every change you make is saved automatically to localStorage — the browser's built-in storage on your device. The header shows a 💾 HH:MM:SS timestamp each time this happens.

    ⚠ localStorage is not permanent backup. It is lost if you clear browser history, open the file from a different folder, or use a different device.

    Sessions & Export tab — three things in one place

    The 💾 Sessions & Export tab handles all data persistence and sharing. It has three sections:

    • Save & Share Sessions — name and save a checkpoint to the browser, download a .json backup file, or load a .json file from a colleague.
    • Saved Sessions list — all named checkpoints stored in the browser on this device. Load, download individually, or delete.
    • Export Farm Data for Research — download the active project's farm data as CSV (for Excel/R/Python) or JSON (for programmatic analysis). A live status panel shows which farms have results ready to export.

    Session .json vs Export files — what is the difference?

    Session .jsonExport CSV / JSON
    ContainsAll projects + all farms + all parametersActive project farms + results only
    PurposeBackup, restore, share full working stateResearch analysis, statistics, database
    Can reload into tool?✅ Yes — via Load from file❌ Not intended for that
    Opens in Excel?❌ Not directly✅ CSV opens directly in Excel

    Recommended practice

    • At the end of each field day: save a named session and download the .json
    • Name sessions descriptively: Mutoko_2025-11-21
    • Before sharing with a colleague: download a session .json and send the file
    • When starting on a new device: load from file using yesterday's .json
    • After running analysis on all farms: download CSV for the research team

    3 · Step-by-Step Workflow

    • Create a Project

      Go to 📁 Projects → enter a name and description → Create Project. Click ⚙ Edit to open the Parameters panel.

    • Define the Calendar

      Parameters → 📅 Calendar. Add periods (e.g. 4 periods of 92/90/91/92 days). Total must equal exactly 365.

    • Define Crop Products

      Parameters → 🌽 Crop Products. Add every product that farms in this project grow, consume, sell, or buy. Enter selling and buying prices per period.

    • Define Member Types

      Parameters → 👥 Member Types. Add household member categories (adult male, adult female, child...). Set labour capacity (days/year) and consumption needs per crop product.

    • Set Hired Labour Price

      Parameters → 👷 Labour Price. Enter the daily wage for hired labour in USD/day.

    • Define Cropping Systems

      Parameters → 🌱 Cropping Systems. For each system add outputs (yield kg/ha/period per product), labour tasks (days/ha/period), and inputs (USD/ha/period).

    • Create Farm Profiles

      Go to 🌾 Farms. Enter Farm ID, name, village → Create Profile. Then click Edit to go to data entry.

    • Enter Farm Data

      📋 Data Entry: fill in household composition, farm area, non-farm income sources, and the crop plan (which cropping systems, how many ha each).

    • Run the Analysis

      Click ▶ Run Analysis. Results appear in 📊 Results: resilience badges, KPIs, cash flow chart, food balance per product, labour table.

    • Run a Scenario

      Go to 🔀 Scenarios. Define an alternative crop plan and click ▶ Run Scenario to see side-by-side comparison with the baseline.

    • Save & Export

      💾 Sessions & Export → give the session a name → Save to Browser + Download .json (your backup). Then scroll down to Export Farm Data for Research → Download CSV to send to the research team.

    4 · Projects & Parameters

    A Project defines the shared reference world for a set of farms. All parameter values — yields, prices, labour requirements — come from the active project. One project is active at a time; switching projects changes the context for Farms and Data Entry.

    Calendar

    The model operates on a year of exactly 365 days divided into user-defined periods. Period names (e.g. P1, P2, Planting, Harvest) are used throughout the interface and in all charts. The number of periods is flexible — a typical Zimbabwe setup uses 4 periods.

    Crop Products

    Any product that can be grown, consumed, sold or bought. Selling price and buying price are defined per period, allowing seasonal price variation. The buying price is used when a household's production falls short of its consumption needs.

    Household Member Types

    Member categories carry two quantities: labour capacity (days/year, prorated to each period by its length) and consumption needs per crop product (kg/year). A household's total consumption need is derived from the sum across all members of each type.

    Cropping Systems

    A cropping system is a complete management package. It defines:

    • Outputs — which crop products are produced, in what quantities (kg/ha) in each period
    • Labour tasks — named operations with labour requirements (days/ha) per period
    • Inputs — named inputs with monetary costs (USD/ha) per period

    This structure replaces the fixed good-year/bad-year and Ext/Int categories from earlier versions. Each scenario corresponds to a named cropping system defined by the researcher.

    5 · Farms & Data Entry

    Household Composition

    For each member type defined in the project, enter the count for this household. This drives both total labour supply and total food consumption needs.

    Farm Area

    Total land area available. The land allocation bar in the crop plan section shows the sum of all plot areas against this total in real time. The analysis is blocked if total cropped area exceeds farm area.

    Non-Farm Income Sources

    Any income not derived from crop sales — wages, remittances, petty trade. Entered as USD amounts per period. Multiple sources can be added. These contribute directly to household cash flow.

    Crop Plan

    A list of cropping system + area (ha) pairs. One farm can have several cropping systems on different plots. The same cropping system can appear more than once (e.g. two separate plots under the same management).

    6 · Model Equations & Calculations

    The engine runs entirely in the browser (see js-engine block). All calculations are per-period unless stated otherwise. i = period index (0 to N−1), cs = cropping system, cp = crop product, mt = member type, row = one line of the crop plan.

    6.1 · Labour Supply

    Available household labour in each period, prorated by the number of days in that period:

    HH_supply[i] = Σmt ( count[mt] × labour_days[mt] × (days[i] / 365) ) // Summed over all member types in the household
    VariableUnitSource
    count[mt]personsHousehold composition (Data Entry)
    labour_days[mt]days/yearMember type definition (Parameters)
    days[i]daysCalendar period length (Parameters)

    6.2 · Labour Demand & Hired Labour

    Total labour required by the crop plan in each period:

    labour_demand[i] = Σrow Σtask ( labour_ha[task,i] × area_ha[row] ) // Summed over all crop plan rows and all tasks in each cropping system hired_days[i] = max( 0, labour_demand[i] − HH_supply[i] ) hired_cost[i] = hired_days[i] × hired_labour_price
    VariableUnitSource
    labour_ha[task,i]days/haCropping system task definition
    area_ha[row]haCrop plan row
    hired_labour_priceUSD/dayParameters → Labour Price

    6.3 · Input Costs

    input_cost[i] = Σrow Σinp ( cost_ha[inp,i] × area_ha[row] ) // Summed over all crop plan rows and all inputs in each cropping system

    6.4 · Production

    Quantity of each crop product produced per period:

    prod[cp,i] = Σrow ( yield_ha[cs(row),cp,i] × area_ha[row] ) // Only rows whose cropping system defines an output for cp tot_prod[cp] = Σi prod[cp,i] // Total annual production across all periods

    6.5 · Household Food Need

    hh_need[cp] = Σmt ( count[mt] × kg_year[mt,cp] ) // Annual consumption need for each crop product, summed over all member types
    VariableUnitSource
    kg_year[mt,cp]kg/yearMember type consumption need (Parameters)

    6.6 · Food Balance & Trade

    Annual balance determines whether the household sells surplus or buys deficit. The balance is computed once per year (summing all periods), then distributed across periods for cash flow purposes.

    balance[cp] = tot_prod[cp] − hh_need[cp] // SURPLUS — household sells the excess if balance[cp] ≥ 0: sold_kg[cp] = balance[cp] sell_rev[cp,i] = (prod[cp,i] / tot_prod[cp]) × sold_kg[cp] × sell_price[cp,i] // Revenue distributed proportionally to when production occurs // DEFICIT — household must purchase the shortfall if balance[cp] < 0: purchase_kg[cp] = −balance[cp] buy_cost[cp,i] = (purchase_kg[cp] / N) × buy_price[cp,i] // Purchases spread equally across all periods
    VariableUnitSource
    sell_price[cp,i]USD/kgCrop product selling price per period
    buy_price[cp,i]USD/kgCrop product buying price per period
    N—Number of periods in the calendar

    6.7 · Cash Flow

    cash_in[i] = Σcp sell_rev[cp,i] + nfi[i] cash_out[i] = input_cost[i] + hired_cost[i] + Σcp buy_cost[cp,i] cash_net[i] = cash_in[i] − cash_out[i] cash_cum[i] = Σj≤i cash_net[j] // Cumulative balance at end of period i end_cash = cash_cum[N−1] // Final annual cash balance
    VariableUnitSource
    nfi[i]USDSum of all non-farm income sources for period i

    6.8 · Gross Farm Margin (GFM)

    Total crop revenue minus all farm-related costs, excluding non-farm income:

    GFM = Σi sell_rev[i] − Σi input_cost[i] − Σi hired_cost[i] // sell_rev[i] = sum across all crop products for period i

    6.9 · Gross Margin per Cropping System (GFM_cs)

    Attribution of GFM to individual cropping systems, used in the Gross Margin by Cropping System chart:

    share[row] = area_ha[row] / Σ{same cs} area_ha vqp[cs,row] = Σcp sell_rev_total[cp] × share[row] inpC[cs,row] = Σinp,i cost_ha[inp,i] × area_ha[row] labC[cs,row] = Σtask,i labour_ha[task,i] × area_ha[row] × hired_fraction[i] × hired_labour_price GFM_cs[cs] = Σrows with cs ( vqp − inpC − labC ) // hired_fraction[i] = max(0, (demand[i]−supply[i]) / demand[i])

    6.10 · Household Income Aggregates

    GM = GFM + Σi nfi[i] // Gross Margin = farm + non-farm HHS = Σmt count[mt] // Total household size inc_pc = GM / HHS // Income per capita

    6.11 · Resilience Classification

    food_secure = ∀ cp with hh_need[cp] > 0 : balance[cp] ≥ 0 econ_secure = end_cash ≥ 0 resilience: food_secure AND econ_secure → "Secure" food_secure XOR econ_secure → "Fragile" NOT food_secure AND NOT econ_secure → "Vulnerable"

    7 · Reading Results

    Resilience badges

    ● Secure Both food balance and end-year cash are non-negative.
    ● Fragile One of the two is in deficit.
    ● Vulnerable Both food and cash are in deficit.

    KPI cards

    • Gross Farm Margin — crop revenue minus input and hired labour costs
    • Total HH Income — GFM plus all non-farm income
    • Income / Capita — Total HH Income divided by household size
    • End-Year Cash — cumulative cash balance after all periods; negative = deficit
    • Hired Labour Cost — total hired labour expenditure across all periods
    • Food Deficits — number of crop products where annual production < household need

    Cash Flow chart

    Bars show income and expenses per period. The line shows the cumulative cash balance. A line crossing below zero indicates a period where the household runs out of cash.

    Food balance section

    One chart per crop product that has either production or a consumption need. Shows production per period (bars) against an even split of annual need (line). The summary row shows total production, annual need, and the balance in kg.

    Labour table

    Shows household labour supply, total crop plan demand, resulting hired labour days, and hired labour cost for each period.

    8 · Scenarios & Comparison

    A scenario replaces the crop plan while keeping everything else (household composition, farm area, non-farm income) identical to the baseline. The same calculation engine runs on the scenario crop plan and produces a parallel set of results.

    Comparison table

    Shows baseline vs scenario for: GFM, Total HH Income, Income/Capita, End-Year Cash, Hired Labour Cost. The Change column is green for improvements and red for deterioration.

    Comparison charts

    Two side-by-side charts for cash flow (cumulative) and gross margin by cropping system. Food balance charts show both baseline and scenario for each product.

    The scenario is identified by its crop plan rows. To simulate agroecological intensification, define a more intensive cropping system in Parameters, then assign it in the scenario crop plan instead of the baseline system.

    9 · Sessions & Export

    Everything related to saving and sharing your work lives in the 💾 Sessions & Export tab.

    Saving a session

    Type a name in the Session Name field and click 💾 Save to Browser. This creates a named snapshot of all projects and farms, visible in the Saved Sessions list below. You can save as many checkpoints as you like — e.g. one per field day.

    Downloading a session (.json)

    Click ⬇ Download .json to save the current state as a file on your device. This is your real permanent backup. The file contains everything: all projects, all farms, all parameters, all results. Send it to a colleague and they can load it with ⬆ Load from file.

    Exporting research data

    Scroll down to the Export Farm Data for Research section. The status panel shows which farms in the active project have results calculated (green ✓) and which are still pending. Run the analysis in 📋 Data Entry for any pending farms before exporting.

    Click ⬇ Download CSV to get a flat file ready for Excel, R, or Python. The CSV includes one row per farm with the following columns:

    • Farm identification: ID, name, village, project name
    • Household composition: total size + one column per member type
    • Farm area and cropped area
    • Cropping systems used (name:area pairs)
    • Income indicators: GFM, total income, income per capita, end-year cash
    • Labour: total demand, total hired days, total hired cost
    • Food balance per crop product: production, need, balance, purchases (kg)
    • Cash flow per period: income, expenses, cumulative balance
    • Resilience status: food secure, econ secure, overall resilience

    Click ⬇ Download JSON for a structured export of the same data — useful for programmatic analysis or database import. This file is not designed to be reloaded into the tool (use a session .json for that).

    10 · Constraints & Validation

    The tool checks the following before allowing an analysis to run. Red errors block the run.

    CheckRuleHow to fix
    Land constraintSum of crop plan areas ≤ farm areaReduce plot areas or increase farm area
    Empty crop planAt least one row requiredAdd a cropping system in Data Entry
    Missing CSEvery crop plan row must have a system selectedSelect a cropping system in the dropdown
    CalendarPeriod days must sum to 365Fix in ⚙ Parameters → Calendar

    Labour over-demand is not an error — it simply means hired labour will be needed and is included in the cost calculations.

    11 · Using on Tablet / Phone (Android)

    The tool is designed to work on Android devices in any modern browser (Chrome, Firefox, Edge). Here is how to set it up for field use:

    Option A — Local file (offline, no internet needed)

    • Copy the .html file to the device (USB cable, Bluetooth, or download via a shared link once)
    • Open a file manager app, navigate to the file, and tap it — it opens in the browser
    • Bookmark it for easy access next time
    • ⚠ localStorage is tied to the file path. If you move the file, you start fresh. Keep it in a fixed folder.

    Option B — Shared drive (requires internet on first open, then cached)

    • Upload the file to Google Drive or a shared server
    • Open the shared link in Chrome on the device
    • In Chrome, tap the menu → "Add to Home Screen" to create an app-like icon

    Tips for touch use

    • Buttons and input fields are sized for finger tapping (minimum 34px height)
    • Use landscape orientation on phones for tables and charts
    • The navigation bar scrolls horizontally — swipe left to see all tabs
    • Download sessions via the ⬇ button — files land in the device's Downloads folder
    • Load sessions by tapping ⬆ Load from file and selecting from Downloads
